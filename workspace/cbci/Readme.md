# 타임라인형 뉴스 구현 관련 정리
## 1. 배경/목표
### 배경
#### 현재 PressAI 기사 재구성(현행)  
* 구조: **원본(1) → AI → 재생성 기사(1)**  
* 입력 원본이 1개이며, 결과도 1개의 재구성 기사로 끝나는 형태로 구성<br>
#### 타임라인형 뉴스  
- **타겟: CBC ‘총수家’**  
* 구조: **원본(N) → AI → 재생성 기사(1차 생산 → 2차 생산 → 지속 업데이트)**  
* 핵심: "한번 작성한 기사"가 아닌, 업데이트 버튼을 통해 동일한 인물 페이지에서 **관련 기사의 타임라인이 누적되는 방식**

<br>

### 목표  
- **자동화/재사용성을 고려한 타임라인형 뉴스의 구현, 실현 가능성 검토**  
- 구현 과정에서, 가져올 소스에 대한 API의 가용 여부 확인
 
구성 요소:  
1. **총수 인물 프로필 정보(정적 데이터)**  
2. **인물 관련 기사(연합뉴스 등) 기반의 동적 타임라인(업데이트 누적)**  
3. **대표(요약) 기사/프로필 본문은 버전업되거나, 타임라인 항목이 계속 붙는 형태**<br>
#### 구체화  
1. (총수 카테고리 기준) 총수 인물 정보 수집  
2. 연합뉴스의 소스를 이용, 수집된 총수의 관련도에 따른 기사 생성, 이 때 기사 생성은 PressAI 재구성 로직으로 요약/재생성  
3. 관리자가 업데이트 버튼으로 1, 2의 워크 플로우를 반복해서 타임라인이 누적되는 기사 페이지 구현  

**-> PressAI 부스팅 뉴스 v2의 재구성 로직, Base44를 이용한 프론트엔드 구현**

## 2. 기능 요구사항 정리  
### A. “총수 인물” 데이터 수집/관리(우선순위 1)  
- 인물 기본 정보(예: 이름/생년/가족/직함/회사/프로필 이미지/키워드 별칭)  
- 목적: 이후 기사 크롤링/검색/타임라인의 **기준 엔티티(검색 쿼리/필터)** 로 사용


필드 예시:
- `person_id`, `name_ko`, `name_en`  
- `birth_date`, `roles`(직함), `company`, `profile_image`  
- `aliases`(예: “이재용”, “삼성전자 회장”, “Samsung Electronics chairman” 등)
* 자녀 정보(필요시)

소스:
* 네이버 인물정보
* 위키피디아 인물정보
* 나무위키 인물정보

<br>

### B. “원본 기사 N개” 수집(5년치) + 지속 수집(우선순위 2)  
- 요구: **인물 기준 5년치 기사 크롤링**
- 목적: 타임라인형 뉴스의 안정적 소스 운용이 가능한지 확인, 원본 기사 저장소 구축, 이를 이용한 재구성 기사 지속적 생성

필드 예시:
- `source_id`, `person_id`
- `publisher`, `title`, `body`, `published_at`, `url`, `hash(중복방지)`

소스: 연합뉴스(프리미엄뉴스) 등

> 참고: PressAI 쪽에는 이미 “URL → 제목/본문 추출 → LLM 재구성” 흐름이 있음


<br>

### C. “AI 생성(재구성) + 업데이트(타임라인)” (우선순위 3)  
타임라인 형태 구현의 핵심은 **생성 결과를 1회성으로 끝내지 않고, 업데이트 시점마다 누적 가능한 ‘이벤트’로 만드는것**

구성 요소:
1. **인물 페이지 대표 본문(프로필형 ‘메인 아티클’)**
2. **타임라인 항목들(업데이트 이벤트)**
    - 업데이트 버튼 누를 때마다 소스를 통해 가져오는 “새 기사"


 CBC식 “원본(N) → AI → 타임라인”을 위한 접근:

설계:  
- “업데이트”는 **새로 들어온 원본 기사들만** 대상으로 처리해야 함
- 처리된 원본은 “사용됨(Consumed)” 상태가 되어야 재처리를 방지
- 타임라인은 “날짜/이슈 단위”로 **클러스터링**해서 중복을 방지할 필요가 있음
